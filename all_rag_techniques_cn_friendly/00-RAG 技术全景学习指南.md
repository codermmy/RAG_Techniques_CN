## 🎯 引言：为什么需要学习 RAG 技术？

在当今的大语言模型（LLM）应用中，**检索增强生成（Retrieval-Augmented Generation，简称 RAG）** 已成为构建生产级 AI 系统的核心技术范式。它将外部知识库与生成式 AI 相结合，让模型能够"先查找，再回答"，有效解决了 LLM 的幻觉问题、知识时效性限制和领域专业性不足等核心痛点。

然而，**基础 RAG 存在明显的局限性**：
- **检索精度不足**：简单的向量相似度匹配常常无法捕捉复杂的语义关系
- **上下文碎片化**：机械的分块策略可能切断原本连贯的语义
- **查询理解浅层**：用户原始查询往往不够精确，直接检索效果有限
- **缺乏推理能力**：无法处理需要多步推理或跨文档综合的复杂问题

本学习指南基于 `all_rag_techniques_cn_friendly` 目录中的丰富实践，按照 RAG 系统的实际处理链路，系统性地讲解**30+ 种主流 RAG 技术**。你将理解每项技术的核心思想、适用场景以及它们在整个数据流中的位置。

## 一、基础篇 🌱

### 1.1 基础 RAG（Simple RAG）

**基础 RAG 是整个技术体系的起点**，它实现了最核心的"检索 - 生成"范式。

**核心流程：**
```
用户查询 → 向量化 → 向量库相似度检索 → 返回 Top-K 文档块 → LLM 生成答案
```

**工作原理：**
1. **文档预处理**：将 PDF、TXT 等格式的文档加载并转换为纯文本
2. **文本分块**：使用固定长度（如 1000 字符）或递归字符分块器将长文档切分为小块
3. **向量编码**：通过 OpenAI Embeddings 等模型将每个文本块转换为向量表示
4. **建立索引**：使用 FAISS 等向量数据库存储和索引这些向量
5. **检索生成**：查询时计算查询向量与文档向量的相似度，返回最相关的上下文供 LLM 生成

**局限性分析：**
- **机械分块破坏语义**：固定长度分块可能在句子或段落中间切断，导致上下文断裂
- **单一检索模式**：仅依赖向量相似度，可能遗漏关键词匹配精确的结果
- **缺乏查询优化**：用户原始查询直接用于检索，没有经过任何增强或改写
- **无后处理机制**：检索到的结果直接使用，没有二次筛选或精炼

> 💡 **学习建议**：先动手实现一个基础 RAG 系统，亲身体验这些局限性，再学习后续的高级技术，你会更深刻地理解每项技术的价值。

**📖 对应文档**：`02-simple_rag-基础 RAG-新手友好版.md`

### 1.2 可靠 RAG（Reliable RAG）

可靠 RAG 在基础 RAG 之上增加了**质量保障机制**，使系统输出更加可信。

**核心增强：**
- **引用溯源**：生成的每个答案都附带明确的来源引用
- **置信度评估**：对检索结果的质量和相关性进行评分
- **答案验证**：检测生成内容与检索上下文的一致性，防止幻觉

**典型实现：**
```python
# 伪代码示例
def reliable_rag(query):
    contexts = retrieve(query, k=5)
    scored_contexts = score_relevance(contexts, query)  # 相关性评分
    filtered = filter_by_threshold(scored_contexts, 0.7)  # 过滤低质量
    answer, citations = generate_with_citations(filtered)  # 带引用生成
    verified = verify_consistency(answer, filtered)  # 一致性验证
    return answer if verified else "无法确定答案，建议人工审核"
```

**📖 对应文档**：`04-reliable_rag-可靠 RAG-新手友好版.md`

### 1.3 RAG 评估机制

**没有评估就没有改进**。RAG 评估是整个技术栈中不可或缺的一环。

**主流评估工具：**

| 工具 | 评估维度 | 特点 |
|------|----------|------|
| **Ragas** | 忠实度、答案相关性、上下文召回率 | 无需标注数据，基于 LLM 自评 |
| **DeepEval** | 答案准确性、上下文精确度、偏见检测 | 支持自定义测试用例 |
| **TruLens** | 三元组评估（上下文 - 答案 - 反馈） | 支持人类反馈集成 |
| **GroUSE** | 基于 GPT-4 的综合评估 | 高准确度，成本较高 |

**核心评估指标：**
- **忠实度（Faithfulness）**：答案是否完全基于检索的上下文，无幻觉
- **答案相关性（Answer Relevance）**：答案是否直接回应了用户问题
- **上下文召回率（Context Recall）**：检索的上下文是否包含回答问题所需的关键信息

**📖 对应文档**：`evaluation/` 目录中的评估 Notebook

> **数据质量决定系统上限**。本阶段的技术专注于如何更好地组织和表示知识。

---

### 2.1 语义分块（Semantic Chunking）

**语义分块解决了"在哪里切分"这个关键问题。**

**核心思想：**
不是按照固定字符数切分，而是**在语义边界处自然分割**，确保每个块内的内容在主题和语境上是连贯的。

**实现原理：**
1. 将文档按句子拆分
2. 计算相邻句子之间的嵌入相似度
3. 当相似度低于某个阈值（如第 90 百分位）时，认为存在语义边界，在此处切分

```
原文：[句子 1][句子 2][句子 3] ||| [句子 4][句子 5][句子 6][句子 7] ||| [句子 8]...
                          ↑ 语义边界                   ↑ 语义边界
分块结果：[块 1: 句子 1-3]  [块 2: 句子 4-7]  [块 3: 句子 8...]
```

**优势：**
- 保持上下文的完整性和连贯性
- 避免因机械切分导致的关键信息断裂
- 检索到的块更有可能是"完整的答案单元"

**适用场景：** 学术论文、法律文档、技术手册等结构严谨的长文档

**📖 对应文档**：`13-semantic_chunking-语义分块.md`

### 2.2 命题分块（Proposition Chunking）

**命题分块是语义分块的进阶版本**，它追求更细粒度的语义完整性。

**核心思想：**
将文本分解为**独立、完整的事实陈述单元**（即"命题"），每个命题表达一个完整的语义。

**实现方式：**
使用 LLM 对文档块进行分析，提取出类似知识图谱三元组结构的陈述：
```
原文："苹果公司于 1976 年由史蒂夫·乔布斯和史蒂夫·沃兹尼亚克在加利福尼亚州创立。"

提取的命题：
- 命题 1：苹果公司的成立时间是 1976 年
- 命题 2：苹果公司的创始人包括史蒂夫·乔布斯
- 命题 3：苹果公司的创始人包括史蒂夫·沃兹尼亚克
- 命题 4：苹果公司的创立地点是加利福尼亚州
```

**优势：**
- 每个命题都是独立可理解的语义单元
- 便于精确检索和事实验证
- 支持细粒度的引用和溯源

**📖 对应文档**：`06-proposition_chunking-命题分块 - 新手入门.md`

### 2.3 父子文档检索（Parent-Child Retriever）

**父子检索解决了"检索粒度"的权衡问题。**

**核心思想：**
- **子块（Child）**：较小的文本块（如 200-500 字），用于精确检索匹配
- **父块（Parent）**：包含子块的较大上下文（如 2000-5000 字），用于提供给 LLM 生成

**工作流程：**
```
1. 检索阶段：使用子块进行向量相似度搜索 → 找到最相关的精确片段
2. 扩展阶段：根据子块的 parent_id 找到对应的父块
3. 生成阶段：将父块（而非子块）作为上下文提供给 LLM
```

**优势：**
- 检索精确：小子块更容易与查询向量精确匹配
- 上下文丰富：父块提供完整的背景信息，避免断章取义
- 两全其美：同时获得了精确检索和丰富上下文的好处

**实现提示：** LangChain 的 `ParentDocumentRetriever` 可直接使用。

### 2.4 假设性问题生成（Questions Contextualization）

**这是一种"主动索引"策略**，在索引阶段为文档块生成可能的问题。

**核心思想：**
对每个文档块，使用 LLM 生成多个"这个文档块可能回答的问题"，将这些问题也嵌入并索引。

```
索引阶段：
文档块 D → LLM 生成 → [问题 Q1, 问题 Q2, 问题 Q3...]
                ↓
         索引：D 的嵌入 + Q1/Q2/Q3 的嵌入

检索阶段：
用户查询 Q → 匹配 Q1/Q2/Q3 → 返回对应的 D
```

**优势：**
- 问题的表述方式更接近真实用户查询
- 桥接了"文档语言"和"用户语言"之间的鸿沟
- 特别适用于 FAQ 场景和客服问答系统

---

### 2.5 树状/层级索引（RAPTOR）

**RAPTOR（Recursive Abstractive Processing and Thematic Organization for Retrieval）** 是一种创新的多层级检索架构。

**核心思想：**
对文档进行**递归摘要**，构建一棵从具体到抽象的树状结构。

```
Level 3:              [整篇文档的摘要]
                     /              \
Level 2:      [章节组摘要 1]    [章节组摘要 2]
               /    \            /    \
Level 1:   [段落组 1] [段落组 2] [段落组 3] [段落组 4]
           /  \      /  \       /  \      /  \
Level 0: [句 1][句 2][句 3][句 4][句 5][句 6][句 7][句 8]
         (原始文档片段)
```

**检索策略：**
- **自顶向下**：从高层摘要开始，逐步向下追踪到具体细节
- **混合层级检索**：同时从多个层级检索相关信息，然后整合

**优势：**
- 支持宏观概览和微观细节的灵活检索
- 能够回答需要跨段落综合的问题
- 对超长文档（如整本书）的处理尤其有效

**📖 对应文档**：`26-raptor-RAPTOR_新手版.md`

### 2.6 上下文分块标题（Contextual Chunk Headers）

**这是一种简单但极其有效的上下文增强技术。**

**核心思想：**
在每个分块前添加其所属的**章节标题或文档元信息**作为"前缀"。

```
原始分块：
"......实验结果表明，该方法准确率提升了 15%......"

添加标题后：
"文档：《深度学习在医学影像中的应用》
章节：3.2 实验结果分析
内容：......实验结果表明，该方法准确率提升了 15%......"
```

**优势：**
- 为分块提供了明确的上下文定位
- 避免分块脱离大背景后被误解
- 实现成本极低，几乎不需要额外计算资源

**📖 对应文档**：`10-contextual_chunk_headers-上下文分块头部 - 新手友好版.md`

### 2.7 上下文窗口增强（Window Around Chunk）

**解决"retrieved 内容太多/太少"的问题。**

**核心思想：**
检索到核心分块后，动态扩展其前后相邻的内容，形成一个"上下文窗口"。

```
原始分块：[......第 i 块......]
扩展后：  [......第 i-1 块......][......第 i 块......][......第 i+1 块......]
```

**实现方式：**
- **固定窗口**：始终扩展前后各 N 个字符或 M 个分块
- **动态窗口**：根据查询复杂度或分块位置动态调整窗口大小

**优势：**
- 提供必要的上下文衔接
- 避免因分块边界切断而丢失关键信息
- 可在检索精度和上下文完整性之间灵活平衡

**📖 对应文档**：`12-context_enrichment_window_around_chunk-上下文窗口增强 - 新手友好版.md`

## 三、查询转化层 🔍

> **查询质量决定检索质量**。本阶段的技术专注于在检索前优化用户查询。

---

### 3.1 查询重写（Query Rewriting）

**查询重写是提升检索效果的第一道关卡。**

**核心思想：**
将用户的原始查询改写为**更清晰、更具体、更适合检索**的形式。

**常见改写策略：**

| 策略 | 示例 |
|------|------|
| **去口语化** | "那个谁说的挺有名的话" → "名人名言" |
| **补充上下文** | "它的首都" → "法国首都"（结合对话历史） |
| **扩展同义词** | "汽车" → "汽车 OR 车辆 OR 机动车" |
| **结构化改写** | "怎么学 Python" → "Python 学习方法 OR Python 学习路线" |

**实现方式：**
- 使用规则模板（适合特定领域）
- 使用 LLM 进行语义改写（更灵活通用）

**📖 对应文档**：`07-query_transformations-查询转换 - 新手入门.md`

---

### 3.2 HyDE（Hypothetical Document Embedding，假设文档嵌入）

**HyDE 是一种"以答案找答案"的巧妙方法。**

**核心思想：**
先让 LLM 生成一个**假设性的答案文档**，然后用这个假设答案的嵌入向量去检索真实文档。

```
用户查询："气候变化的主要原因是什么？"
   ↓
LLM 生成假设答案："气候变化的主要原因是人类活动导致的温室气体排放增加，包括化石燃料燃烧、 deforestation 等......"
   ↓
将假设答案嵌入为向量 → 检索真实文档
```

**为什么有效？**
- 假设答案的语义空间与真实答案更接近
- 解决了查询简短而文档冗长之间的"语义鸿沟"
- 尤其适用于开放性问题

**注意事项：**
- 假设答案不需要准确，只需要在语义空间上接近真实答案
- 即使 LLM"胡说八道"，其嵌入向量仍可能有效指引检索方向

**📖 对应文档**：`08-HyDe_Hypothetical_Document_Embedding-假设性文档嵌入 - 新手入门.md`

---

### 3.3 HyPE（Hypothetical Prompt Embeddings，假设提示嵌入）

**HyPE 是 HyDE 的进阶变体。**

**核心思想：**
不仅生成假设答案，还通过精心设计的提示词让 LLM 生成**多个角度的假设性响应**，然后综合这些响应的嵌入。

```
用户查询 →
├─ 提示 1："请作为一个专家回答..." → 嵌入 1
├─ 提示 2："请用通俗语言解释..." → 嵌入 2
├─ 提示 3："请列出关键点..." → 嵌入 3
   ↓
融合嵌入 1/2/3 → 检索
```

**优势：**
- 多视角覆盖，增强检索的召回率
- 适用于需要综合多方面信息的复杂查询

**📖 对应文档**：`09-HyPE_Hypothetical_Prompt_Embeddings-假设性提示嵌入 - 新手入门.md`

### 3.4 多路查询与 RAG-Fusion

**多路检索 + 结果融合 = 更全面的召回。**

**核心思想：**
1. 生成原始查询的多个变体
2. 用每个变体独立检索
3. 使用**倒数排名融合（RRF, Reciprocal Rank Fusion）** 合并结果

```
原始查询："可再生能源的优势"
   ↓
变体 1: "可再生能源好处" → 结果集 A
变体 2: "太阳能风能优点" → 结果集 B
变体 3: "清洁能源优势" → 结果集 C
   ↓
RRF 融合 → 最终排序结果
```

**RRF 算法核心：**
对每个文档，计算其在各结果集中的排名倒数和：
```
RRF 得分 = Σ (1 / (k + 排名_i))
```

**优势：**
- 减少单一查询的偏差和遗漏
- 提升召回率，特别适用于信息密集型问题

**📖 对应文档**：`16-fusion_retrieval-融合检索.md`

### 3.5 退后提示（Step-back Prompting）

**退后一步，从更高层面理解问题。**

**核心思想：**
先让 LLM 提取查询背后的**核心概念或原理**（"退后"），然后用这个抽象概念辅助检索。

```
用户查询："光合作用中 ATP 是如何生成的？"
   ↓
退后问题："请解释光合作用的基本原理和能量转换过程"
   ↓
检索 → 获取背景知识
   ↓
结合原始查询和检索结果 → 生成精确答案
```

**适用场景：**
- 需要背景知识才能精确理解的查询
- 涉及复杂概念的问题
- 教育、科普类问答系统

---

### 3.6 语义路由（Semantic Router）

**根据查询类型，智能选择处理路径。**

**核心思想：**
使用轻量级模型判断查询的**意图类型**，然后路由到最适合的检索策略或知识库。

```
用户查询
   ↓
语义分类器 → "这是技术问题"
   ↓
路由到 → 技术文档检索管道

用户查询
   ↓
语义分类器 → "这是闲聊"
   ↓
路由到 → 通用对话模型（跳过检索）
```

**常见路由类型：**
- **检索 vs 不检索**：判断是否需要外部知识
- **领域路由**：技术/医疗/法律/通用
- **难度路由**：简单事实查询 → 直接检索；复杂推理 → 多步检索或 Agent 处理

**优势：**
- 降低不必要的检索开销
- 提升专业领域问题的回答质量
- 实现成本远低于全量 Agent 系统

---

## 四、检索与后处理层 🚀

> **检索不是终点，而是精准答案的起点**。本阶段技术专注于优化检索结果。

---

### 4.1 混合检索（Hybrid Search）

**混合检索是工业级 RAG 系统的标配。**

**核心思想：**
同时进行**向量检索（语义相似度）**和**关键词检索（如 BM25）**，然后融合结果。

```
用户查询 →
├─ 向量检索 → 捕捉语义相似 → 结果 A
├─ BM25 检索 → 捕捉关键词匹配 → 结果 B
   ↓
融合（RRF/加权） → 最终结果
```

**为什么需要混合？**
| 检索类型 | 擅长 | 不擅长 |
|----------|------|--------|
| 向量检索 | 语义相似、同义词 | 专有名词、精确匹配 |
| 关键词检索 | 术语、人名、代码 | 语义等价、改写 |

**实现建议：**
- LangChain 的 `EnsembleRetriever`
- LlamaIndex 的 `ReciprocalRerankFuseRetriever`
- 向量数据库如 Weaviate、Milvus 原生支持混合检索

**📖 对应文档**：`16-fusion_retrieval-融合检索.md`

---

### 4.2 元数据过滤（Metadata Filtering）

**用结构化信息约束检索范围。**

**核心思想：**
在向量检索的基础上，添加**元数据条件过滤**（如时间、作者、文档类型等）。

```
查询："2023 年之后发表的关于深度学习的综述"
   ↓
过滤条件：{ "年份": >=2023, "类型": "综述", "关键词": "深度学习" }
   ↓
在过滤后的子集中进行向量检索
```

**常见元数据：**
- 时间戳（发布日期、更新时间）
- 来源（作者、机构、网站）
- 类型（论文、新闻、博客、代码）
- 主题分类
- 语言、地区

**优势：**
- 大幅提升检索精度
- 支持时效性敏感的应用
- 可实现细粒度的权限控制

---

### 4.3 重排序（Reranking）

**重排序是提升检索质量的"最后一道关卡"。**

**核心思想：**
先用快速但粗糙的检索方法（如向量检索）获取候选集（如 Top-50），然后用更精确但计算量大的模型重新排序，选择最终的 Top-K。

```
第一阶段（粗排）：
查询 → 向量检索 → Top-50 候选文档（快速，毫秒级）

第二阶段（精排）：
查询 + Top-50 → Cross-Encoder 重排序 → Top-5 最终结果（较慢，百毫秒级）
```

**为什么有效？**
- **向量检索（Bi-Encoder）**：查询和文档独立编码，计算余弦相似度，速度快但精度有限
- **重排序（Cross-Encoder）**：查询和文档一起输入模型，进行深度交互，精度高但计算量大

**常用重排序模型：**
- Cohere Rerank
- BGE Reranker
- ColBERT
- LLM-based Reranking（使用 LLM 直接评分）

**投入产出比：**
重排序通常是提升 RAG 系统效果**最立竿见影**的技术，建议优先考虑。

**📖 对应文档**：`17-reranking-重排序.md`

---

### 4.4 上下文压缩（Contextual Compression）

**去粗取精，只保留最相关的部分。**

**核心思想：**
对检索到的文档进行**压缩和提炼**，去除与查询无关的内容，只保留核心信息。

```
检索到的文档（500 字）：
"......（背景介绍 200 字）......（无关细节 150 字）......（核心答案 150 字）......"
   ↓
LLM 压缩 →
"核心答案 150 字"
   ↓
提供给生成模型
```

**实现方式：**
- **提取式压缩**：用 LLM 识别并提取与查询相关的句子（如 LangChain 的 `LLMChainExtractor`）
- **摘要式压缩**：对相关段落进行摘要改写

**优势：**
- 减少无关噪声对 LLM 的干扰
- 节省 Token 成本
- 提升答案的精确度和可追溯性

**📖 对应文档**：`14-contextual_compression-上下文压缩.md`

---

### 4.5 相关片段提取（Relevant Segment Extraction, RSE）

**RSE 是上下文压缩的精细化版本。**

**核心思想：**
不仅压缩单个文档，还跨多个检索结果**识别并提取最相关片段**，然后组装成连贯的上下文。

```
检索结果 1: [无关][相关片段 A][无关]
检索结果 2: [相关片段 B][无关][相关片段 C]
检索结果 3: [无关][相关片段 D]
   ↓
提取并组装 → [A][B][C][D] → 生成
```

**优势：**
- 最大化利用有限的上下文窗口
- 避免冗余和重复信息
- 适用于需要综合多来源的复杂问题

**📖 对应文档**：`11-relevant_segment_extraction-相关片段提取 - 新手友好版.md`

---

### 4.6 飞镖盘检索（Dartboard Retrieval）

**这是一种较新的检索策略**，核心思想是将查询分解为多个"靶心"，分别检索后整合。

**核心思想：**
1. 将复杂查询分解为多个子问题（靶心）
2. 对每个子问题独立检索
3. 整合所有结果，形成完整答案

**适用场景：**
- 多维度综合问题
- 需要对比分析的场景
- 调研报告生成

**📖 对应文档**：`19-dartboard-飞镖板检索 - 新手入门.md`

---

## 五、高阶架构层 🏗️

> **从"检索工具"到"智能系统"**。本阶段技术引入反思、纠错和代理能力。

---

### 5.1 Self-RAG（自反思 RAG）

**Self-RAG 让系统具备"自我评估"能力。**

**核心思想：**
在检索和生成过程中，让 LLM 主动生成**反思标记（Reflection Tokens）**，评估每个步骤的质量。

**工作流程：**
```
1. 检索前反思：「我需要检索吗？」→ 是/否
2. 检索后反思：「检索到的内容相关吗？」→ 相关/部分相关/不相关
3. 生成后反思：「我的回答有依据吗？」→ 完全有依据/部分有依据/无依据
```

**关键创新：**
- 训练 LLM 在输出答案的同时输出反思标记
- 反思标记用于过滤低质量输出或触发重新检索

**优势：**
- 减少幻觉和无关答案
- 动态决定是否需要检索
- 提升答案的可信度

**📖 对应文档**：`27-self_rag-自 RAG_新手版.md`

---

### 5.2 Corrective RAG（CRAG，纠正式 RAG）

**CRAG 在 Self-RAG 的基础上增加了"纠正"机制。**

**核心思想：**
当检索结果质量不佳时，系统能够**主动采取纠正措施**，而不是强行生成答案。

**纠正策略：**
```
评估检索结果质量
   ↓
高质量 → 直接生成答案
中质量 → 生成 + 标注不确定性
低质量 →
├─ 尝试重新检索（换查询词）
├─ 搜索外部知识源
└─ 明确告知用户"无法回答"
```

**关键组件：**
- **质量评估器**：对检索结果进行相关性评分
- **纠正策略选择器**：根据评分决定采取何种纠正措施

**优势：**
- 显著提升答案的可靠性和诚实性
- 避免"硬编答案"的情况
- 适用于对准确性要求高的场景（如医疗、法律）

**📖 对应文档**：`28-crag-校正 RAG-新手友好版.md`

---

### 5.3 Agentic RAG（代理式 RAG）

**Agentic RAG 将 RAG 系统升级为具备自主决策能力的智能代理。**

**核心思想：**
使用 LLM Agent 框架（如 LangChain Agent）来**自主规划和执行**多步检索任务。

**Agent 能力：**
```
用户问题 →
├─ 任务规划：「我需要先查 A，再查 B，然后对比 C」
├─ 工具调用：选择并调用合适的检索工具
├─ 结果评估：判断当前信息是否足够
├─ 迭代优化：不足则继续检索或调整策略
└─ 答案综合：整合多步结果生成最终答案
```

**典型工具：**
- 向量检索工具
- 关键词搜索工具
- 网页搜索 API
- 代码执行工具
- 计算器

**适用场景：**
- 复杂多跳推理问题
- 需要实时信息的场景
- 跨多个知识源的综合分析

**📖 对应文档**：`01-Agentic_RAG-Agent 式 RAG 新手友好版.md`

---

### 5.4 Graph RAG（知识图谱 RAG）

**Graph RAG 将结构化知识（图谱）与非结构化文本相结合。**

**核心思想：**
从文档中提取实体和关系构建知识图谱，检索时同时利用图谱的结构化推理能力和文本的丰富语义。

```
文档 → 实体/关系抽取 → 知识图谱
            ↓
查询 → 图谱查询 + 文本检索 → 融合答案
```

**优势：**
- 支持关系推理（如"A 的合作伙伴的子公司"）
- 发现隐含关联
- 提供更结构化的答案呈现

**实现方式：**
- LangChain + Neo4j
- 自定义图谱构建流程

**📖 对应文档**：`24-graph_rag-图 RAG_新手版.md`

---

### 5.5 Microsoft GraphRAG

**微软推出的 GraphRAG 实现**，专注于利用 LLM 自动构建和利用知识图谱。

**核心特点：**
- **自动图谱构建**：LLM 从文档中提取实体、关系和属性
- **社区检测**：对图谱进行聚类，发现"社区"结构
- **社区摘要**：为每个社区生成摘要，支持宏观查询
- **多粒度检索**：支持实体级、关系级、社区级的灵活检索

**适用场景：**
- 大规模文档集的综合分析
- 发现隐藏模式和趋势
- 生成综述性报告

**📖 对应文档**：`25-Microsoft_GraphRag-微软 GraphRAG_新手版.md`

## 六、特殊技术 🌟

---

### 6.1 多模态 RAG（Multi-Modal RAG）

**将 RAG 扩展到图像、图表等多模态内容。**

**核心思路：**
1. **图像描述生成**：用 VLM（视觉语言模型）为图片生成文字描述
2. **统一索引**：将描述文本与文档一起索引
3. **跨模态检索**：查询文本 → 检索图片描述 → 返回对应图片

**进阶方案：**
- 使用 ColPali 等原生多模态嵌入模型
- 直接对图像进行向量化，支持图像到图像的检索

**📖 对应文档**：
- `20-multi_model_rag_with_captioning-多模态 RAG 标注 - 新手入门.md`
- `35-multi_model_rag_with_colpali-多模态 RAG_ColPali 版 - 新手友好版.md`

---

### 6.2 可解释检索（Explainable Retrieval）

**让检索过程透明化。**

**核心思想：**
不仅返回检索结果，还解释**为什么这个结果被认为是相关的**。

**实现方式：**
- 高亮显示与查询匹配的关键句子
- 用 LLM 生成相关性解释
- 可视化相似度得分和排名依据

**价值：**
- 增强用户对系统的信任
- 便于调试和优化检索策略
- 符合监管合规要求

**📖 对应文档**：`23-explainable_retrieval-可解释检索_新手版.md`

---

### 6.3 迭代检索（Iterative Retrieval）

**多轮检索，逐步逼近最佳答案。**

**核心思想：**
根据前一轮检索结果的质量，动态调整查询策略，进行多轮检索。

```
第 1 轮检索 → 评估结果 → 发现信息不足
   ↓
调整查询（补充/改写） → 第 2 轮检索
   ↓
评估 → 信息充足 → 生成答案
```

**适用场景：**
- 复杂问题需要多角度信息
- 初始查询模糊或不精确
- 探索性研究和信息搜集

---

### 6.4 自适应检索（Adaptive Retrieval）

**根据查询特点动态选择检索策略。**

**核心思想：**
系统内置多种检索策略（如向量检索、混合检索、图谱检索等），根据查询的类型、复杂度等特征**自动选择最优策略**。

**决策维度：**
- 查询长度和复杂度
- 领域分类
- 历史效果反馈
- 当前系统负载

**📖 对应文档**：`22-adaptive_retrieval-自适应检索 - 新手入门.md`

---

### 6.5 检索反馈循环（Retrieval with Feedback Loop）

**让系统越用越聪明。**

**核心思想：**
收集用户对检索结果的反馈（显式评分或隐式行为），用于优化后续的检索。

**反馈类型：**
- **显式反馈**：用户对结果的点赞/点踩
- **隐式反馈**：点击、停留时间、引用行为
- **生成结果反馈**：生成答案的质量回溯到检索评估

**优化方式：**
- 调整嵌入模型（微调）
- 更新检索权重参数
- 优化分块策略

**📖 对应文档**：`21-retrieval_with_feedback_loop-反馈循环检索 - 新手入门.md`

## 七、技术选型与组合建议 🎯

没有"最好"的技术，只有**最适合场景**的组合。以下是针对不同场景的建议：

### 场景 1：企业知识库问答

**推荐组合：**
```
基础架构：基础 RAG + 元数据过滤（按部门/权限）
   ↓
增强：语义分块 + 混合检索 + 重排序
   ↓
可选：上下文窗口增强 + 可靠 RAG（引用溯源）
```

### 场景 2：客服问答系统

**推荐组合：**
```
基础架构：查询重写 + 语义路由（区分业务问题/闲聊）
   ↓
增强：假设性问题生成（FAQ 匹配） + HyDE
   ↓
可选：Self-RAG（评估答案质量）
```

### 场景 3：学术研究助手

**推荐组合：**
```
基础架构：RAPTOR 树状索引 + 多路检索融合
   ↓
增强：父子文档检索 + 上下文压缩
   ↓
可选：Graph RAG（发现跨论文关联）
```

### 场景 4：复杂分析任务

**推荐组合：**
```
基础架构：Agentic RAG + 迭代检索
   ↓
增强：多模态 RAG（处理图表） + 重排序
   ↓
可选：CRAG（低质量时主动纠正）
```
